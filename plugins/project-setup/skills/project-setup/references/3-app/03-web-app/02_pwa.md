# PWA — the web frontend, installable

A Progressive Web App is **not a new app**. It is the existing web frontend plus two additions — a web app manifest and a service worker — that make it installable and (optionally) offline-capable. There is no new folder under `apps/`; the whole decision is about what to add *inside* the frontend that already exists.

## PWA vs native mobile vs desktop wrapper

Ask the user what the install target actually needs, then pick the cheapest surface that delivers it:

| Want | Reach for | Owner |
|---|---|---|
| Installable web app, home-screen icon, works offline-ish, one codebase, instant updates | **PWA** (this file) | this file |
| True native UX (gestures, platform widgets), background execution, full push on iOS, app-store presence | **Native mobile** | `references/3-app/07-mobile-app/00_mobile-app.md` |
| Desktop OS integration (tray, menus, filesystem, local daemons) | **Desktop wrapper** | `references/3-app/06-desktop-app/00_desktop-app.md` |

The question to ask: *"Does this need to be in an app store, run in the background, or use device APIs the browser can't reach — or is 'installable web app that mostly works offline' enough?"* If the honest answer is the latter, it's a PWA and you add nothing to the repo shape. PWA and native are **not** mutually exclusive — a product can ship both; see the composition note below.

## Structural rule — it lives in the frontend

The manifest and service worker are frontend assets. They compose with the frontend skeleton in `references/3-app/03-web-app/00_app-skeleton.md`; nothing here creates a sibling app.

- **Manifest** → `<frontend>/public/manifest.webmanifest` (name, icons at 192/512, `display: standalone`, `theme_color`, `start_url`), linked from `index.html`.
- **Service worker** → registered from the app entry, generated by the build (`vite-plugin-pwa` / Workbox) rather than hand-written. Do not commit a hand-rolled `sw.js` you then maintain by hand — let the plugin emit the precache manifest.
- **Icons / offline fallback** → in `public/`, alongside the manifest.

### Caching strategy

- **App shell** (HTML/CSS/JS/fonts) → **precache**. The build hashes assets and the SW serves them cache-first, so the shell opens instantly and offline.
- **Runtime data** (API responses, images) → **runtime caching** with an explicit policy per route: network-first for data that must be fresh, stale-while-revalidate for tolerant data, cache-first only for immutable assets. Never blanket-cache `/api/*`.
- **Updates** → a new SW installs in the background and activates on next load (or via an explicit "refresh to update" prompt). Decide the refresh flow with the user: silent-on-next-visit (default) vs a visible "new version available" toast. Skip-waiting-on-every-load surprises users mid-session — prefer the prompt for apps with in-progress state.

## Installability checklist

- Served over HTTPS (localhost is exempt for dev).
- Linked `manifest.webmanifest` with `name`/`short_name`, `start_url`, `display: standalone`, and maskable icons at 192 + 512.
- A registered service worker with a fetch handler (the shell precache satisfies this).
- Correct `theme_color` / `background_color` so the splash and status bar match the app.
- Passes an install audit (Lighthouse PWA category) before you claim it's installable.

## Offline scope — decide explicitly

Pick one; don't drift into "some things happen to work offline":

1. **None** — installable + fast shell, but requires network for all data. Simplest; honest for data-heavy apps.
2. **Read-only shell** — shell + last-seen data render offline; writes are disabled or queued-with-notice. The common, sane default.
3. **Full offline with sync** — reads and writes work offline, queued mutations reconcile on reconnect (background sync / a local store + replay). Real engineering cost and conflict-resolution design — only commit to it when the product genuinely needs it.

## Push notifications — reality check

Web Push works on Chromium and Firefox. On iOS/Safari it works **only for an installed (home-screen) PWA** on recent iOS, with tighter limits than native, and not at all for a PWA merely opened in the browser tab. If reliable push across iOS is a hard requirement, that is a signal toward **native mobile** (`references/3-app/07-mobile-app/00_mobile-app.md`), not a reason to fight Safari. Treat push as a capability you confirm per-platform, not one you assume.

## Composition with native

When a product ships both a PWA and a native app, they are two delivery surfaces over the **same backend contract**, exactly like the mobile/backend split — the PWA is the web frontend; the native app is its own codebase under `apps/`. Share at the API-contract level, not the code level. See `references/3-app/07-mobile-app/00_mobile-app.md`.

## Anti-patterns

- **Forking the frontend into a "PWA version"** — there is one frontend; the PWA is that frontend with a manifest + SW. A parallel copy rots immediately.
- **Blindly caching API responses** — serving stale user data as if fresh. Every runtime cache needs an explicit, per-route freshness policy.
- **Treating the service worker as a feature** — it's infrastructure. Generate it from the build, keep it dumb, and don't scatter business logic into it.
- **Claiming "offline" without choosing a scope** — decide none / read-only / full-sync up front; half-offline is worse than clearly-online.
- **Assuming push works everywhere** — verify iOS/Safari constraints before promising notifications.
- **Skipping the install audit** — a missing manifest field or non-HTTPS origin silently blocks installability.
