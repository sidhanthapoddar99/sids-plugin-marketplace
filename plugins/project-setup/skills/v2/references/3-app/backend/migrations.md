# Migrations — style choice + run model

Owns the two per-backend migration decisions: **which style** (Alembic autogenerate vs Alembic + raw-SQL shim vs another language's native tool vs no tool) and **which run model** (entrypoint-migrates vs one-shot migrate service vs neutral `apps/db` owner). This file holds the decisions only — every recipe step (init, `env.py`, the shim, the drift check, the entrypoint script) lives in the mechanics files and is linked, never restated here.

## Style choice — how the schema is authored

Default: **Alembic**, for any Python backend over a relational DB. Depart only for a stated reason below.

| Style | Use when | Mechanics |
|---|---|---|
| **Alembic** (autogenerate or hand-write) | Python-only, relational DB (Postgres / SQLite) | `references/3-app/backend/alembic-recipe.md` |
| **Alembic + raw-SQL shim** | A non-Python service (Rust/Go/…) also queries the schema, or DDL is hand-tuned (partitioning, extensions, custom types) and must stay reviewable | `references/3-app/backend/raw-sql-recipe.md` |
| **Another language's native tool** | Python isn't the schema owner — Rust-first, Go-first, or a TS/ORM-owned schema | that tool's own docs |
| **DB-service-managed** | Hasura / Supabase / Atlas / sqitch own the schema | that platform's flow |
| **No tool** | Prototype, schemaless store, or DDL owned by IaC | see below |

### Alembic — the default

Mature, SQLAlchemy-integrated. Within Alembic, one sub-decision — **autogenerate vs hand-write** the revision body:

| Autogenerate when | Hand-write when |
|---|---|
| Simple column add/remove | Partitioning, hash partitions, complex constraints |
| New tables, no special constraints | Data migrations (`UPDATE` statements) |
| Index changes | Extension installs (`CREATE EXTENSION pgvector`), custom types |
| — | Multi-tenant schema setup, anything Alembic can't introspect |

Autogenerated revisions are **always reviewed and edited** before commit — Alembic misses things (e.g. a column rename reads as drop + create). Mixing autogen and hand-write in one revision file is a defect; split them. Command + flow: `references/3-app/backend/alembic-recipe.md`.

### Alembic + raw-SQL shim — multi-language schema consumers

When Rust/Go/another non-Python service queries the same schema, the DDL becomes a **contract**, not a Python implementation detail. Author the schema in raw `.sql` files; Alembic runs each via a 3-line shim. Motivation:

- **Non-Python tooling must read the schema.** Rust's `sqlx` does offline compile-time query checks against the migrated schema; ORM models aren't a contract for it.
- **DBAs / operators review `.sql`,** not `Base.metadata.create_all` output.
- **Hand-tuned DDL** is natural in SQL, verbose in SQLAlchemy.
- **Single source of truth is the `.sql` file;** Python is just the runner.

Use it for: multi-language consumers, complex DDL, DBAs in the loop, strong schema reviewability. **Don't** use it for Python-only projects (plain Alembic is simpler), rarely-changing schemas (overhead not worth it), or ORM-first teams. Mechanics — three-file pattern, shim, helpers, drift check: `references/3-app/backend/raw-sql-recipe.md`.

### Another language's native tool — when Python isn't the owner

If Python is one consumer among several, defer to the language with the strongest migration ecosystem:

- **Rust-first** → `sqlx migrate` with `*.sql` files
- **Go-first** → `golang-migrate/migrate` or `goose`
- **TS/ORM-owned** → Prisma / Drizzle / Kysely, or Diesel (Rust) / Ent (Go)

### DB-service-managed

Some stacks manage schema outside any app: Hasura (schema in Hasura metadata), Supabase (Supabase CLI), or schema-as-code via Atlas / sqitch. Here Alembic would be duplicative and lose features — use the platform's flow.

### No tool

Justified only for genuinely trivial or non-relational cases:

- **Prototype / single-table** → a single `init.sql` mounted at `/docker-entrypoint-initdb.d/01_schema.sql`. Adopt Alembic the moment the schema starts changing.
- **Read-only analytics DB** managed elsewhere, or schema defined as test fixtures.
- **Schemaless / NoSQL** — Mongo (validators in app code), Redis (key conventions), Neo4j/Kuzu (Cypher), Elasticsearch (index mappings via API). Each has its own conventions; don't force Alembic in.
- **ORM-less cloud stores** — DynamoDB (tables via Terraform/CDK), BigQuery/Snowflake (DDL via console or dbt).

Hand-rolled (`migrations/` of numbered SQL + a `schema_migrations` table + a runner script) reinvents Alembic poorly — justified only for a tiny utility that owns one table. For anything growing, use Alembic.

### Decision flow

```
Is the project Python-only with a relational DB?
├── Yes → Alembic (default; autogenerate or hand-write per the table above)
├── No, multi-language over shared Postgres → Alembic + raw-SQL shim
└── No, Python isn't the schema owner → that language's native tool
    ├── Rust → sqlx migrate
    ├── Go   → golang-migrate or goose
    ├── TS   → Prisma / Drizzle / Kysely
    └── Multiple, no obvious owner → raw SQL files + a simple runner
Non-relational / schemaless / IaC-owned → no migration tool (see "No tool")
```

## Run model — who applies migrations, and when

Independent of style: once revisions exist, decide **who runs `upgrade head` and when**. Three models; pick by replica count and backend count, not by habit.

| Run model | Use when | Where the mechanics live |
|---|---|---|
| **Entrypoint-migrates** (container runs `alembic upgrade head` on boot, before the server) | Single-replica backend owns its DB — dev, SQLite / single-Postgres, most deployments | entrypoint script: `references/3-app/backend/alembic-recipe.md` |
| **One-shot migrate service** (a separate job runs `upgrade head` once, before app replicas start) | **N replicas** of one backend — entrypoint-per-replica would race and contend on locks | `references/2-repo/deployment/production-readiness.md` § migrations-on-deploy |
| **Neutral `apps/db` owner** (a standalone migrations app, imports no backend code, run only via `ctl migrate`) | **Two backends share one database** — neither backend may own the DDL | `references/3-app/backend/two-plane-split.md` |

Rules of thumb:

- **Single replica, single backend → entrypoint-migrates.** First `compose up` and the schema just exists; idempotent no-op on every subsequent boot. This is the default.
- **Multi-replica → one-shot.** Same `alembic upgrade head`, different orchestration: run it once as a gate, not once per replica.
- **Two backends, one DB → neutral owner, never on boot.** The migrations move to a standalone `apps/db`; the rationale, structure, and one-DB ownership rules are owned by `references/3-app/backend/two-plane-split.md`.

## What the skill should ask

- What language(s) consume this schema? (Python-only → Alembic; +Rust/Go → raw-SQL shim; Python not the owner → native tool.)
- Is the DDL hand-tuned (partitioning / extensions / custom types) or must a DBA review it? (Yes → raw-SQL shim even if Python-only.)
- How many replicas of the backend run in prod? (1 → entrypoint-migrates; N → one-shot migrate service.)
- Does more than one backend write this database? (Yes → neutral `apps/db` owner — escalate to `two-plane-split.md`.)
- Is the store relational at all? (No → no migration tool; use the engine's conventions.)

## Audit checks

- Python backend on a relational DB with no migration tool and a changing schema → finding (adopt Alembic; retrofitting later is painful).
- `Base.metadata.create_all()` running in production → red finding — migrations must be explicit.
- Two migration systems (Alembic + hand-rolled) against one DB → finding; pick one.
- N-replica backend using entrypoint-migrates → finding — switch to the one-shot migrate service.
- Two backends over one DB where one owns the DDL, or any backend migrating on boot → finding — escalate to `two-plane-split.md` (neutral `apps/db`).
- A recorded migration style/owner in CLAUDE.md that the repo no longer matches → finding.

## Anti-patterns

- Avoiding Alembic because "it's too complex" → adopt earlier, not later.
- Reaching for the raw-SQL shim on a Python-only project → plain Alembic is simpler.
- Letting Rust/Go autogenerate the schema from queries when it's a shared contract → the schema becomes implicit and the DB loses its single owner.
- Migrating away from Alembic with no clear successor in place.
- Every backend racing `upgrade head` on boot at N replicas, or two backends racing it over one DB.
